# T00 — Architecture Overview & Tech Stack

This is the first Technical document. It establishes the system context, technology
choices, and cross-cutting conventions that every later Technical document (T01–T07)
builds on. The stack decisions in §2 and §9 have been reviewed and confirmed.

---

## 1. System context

KaraoGUI is a single-backend, two-frontend, real-time party game. Everything is
scoped to a **Game** (see Business `07-glossary-and-entities.md`).

```
                         ┌──────────────────────────────┐
                         │     Spring Boot Backend       │
                         │  (game engine, source of      │
                         │   truth, REST + WebSocket)    │
                         └───────┬───────────────┬───────┘
                       REST +    │               │  REST +
                     WebSocket   │               │  WebSocket
                 ┌───────────────┘               └───────────────┐
                 ▼                                                 ▼
    ┌─────────────────────────┐                     ┌─────────────────────────┐
    │  TV Frontend (Angular)  │                     │ Phone Frontend (Angular)│
    │  display-only, shared    │                     │ per-player controller   │
    │  big screen              │                     │ role-aware interaction  │
    └─────────────────────────┘                     └─────────────────────────┘
                 ▲                                                 ▲
                 │                                                 │
            shown on a TV                                every player's phone
                                        │
                                        ▼
                         ┌──────────────────────────────┐
                         │   PostgreSQL (containerized)  │
                         │   schema managed by Flyway    │
                         └──────────────────────────────┘
```

- **Backend** is the single **source of truth** and the **authoritative game engine**.
  All rules, timers, scoring, assignment, and state transitions run here — the
  frontends never compute authoritative state (see T04).
- **TV frontend** is **display-only**. It subscribes to a game's live state and
  renders it; it takes no user input.
- **Phone frontend** is the **interactive controller**, one instance per player. What
  it shows and allows depends on the player's **role for the current performance**
  (see Business `01-roles-and-participants.md`).

## 2. Technology stack

### Backend
- **Language / runtime:** Java 25 (LTS; modern language features).
- **Framework:** Spring Boot 4.1 (latest).
- **Web:** Spring MVC for REST (T02).
- **Real-time:** Spring WebSocket with **STOMP** messaging (T03) —
  chosen over Server-Sent Events because the live feed is bidirectional and
  topic-oriented (comments, likes, confirmations, timers, ranking updates) and STOMP
  gives us clean topic/subscription semantics with per-destination authorization.
- **Persistence:** Spring Data JPA / Hibernate over **PostgreSQL**.
- **Schema migrations:** **Flyway** (versioned SQL migrations; the schema is never
  auto-generated by Hibernate in any environment — `ddl-auto` is `validate`).
- **Scheduling:** Spring `@Scheduled` for the recurring Physical-Challenge injection
  job (T05).
- **Build:** Gradle

### Frontends
- **Framework:** Angular (latest stable) + TypeScript, for **both** the TV and Phone
  apps (T06).
- **Real-time client:** STOMP-over-WebSocket client (e.g. `@stomp/stompjs`).
- **Media:** YouTube embeds for karaoke/dance/acting references; image upload for
  player pictures.
- **Structure:** an Nx or Angular-CLI **workspace with two applications**
  (`tv`, `phone`) sharing a common library of models, the real-time client, and API
  clients. This keeps the event/DTO contracts identical across surfaces.

### Data & infrastructure
- **Database:** PostgreSQL, **containerized** (Docker). A `docker-compose` setup
  brings up both PostgreSQL and the backend for local development (one-command
  bring-up).
- **Migrations:** Flyway runs on backend startup and in CI, applying versioned
  migrations from `src/main/resources/db/migration`.

## 3. Identity & sessions (interim)

The Business specs describe no login. For now, identity is **disposable and
per-game**:

- When a person joins a game (by entering the game's **join code** — the short
  `A1A 1A1`-format code shown on the TV; see T01 §2.1a), they provide a **display
  name** and optional **picture**. The backend creates a **Player** scoped to that
  one game and issues a **session token** (an opaque, unguessable identifier) that
  the phone stores locally.
- The **join code** (public, human-typed, identifies the *game*) and the **session
  token** (secret, machine-held, identifies the *player*) are distinct: the code gets
  you to the game, the token authenticates you as a player within it.
- The token identifies the player on subsequent REST calls and WebSocket
  subscriptions for **that game only**. It carries **no cross-game identity** and no
  password.
- Because identity is disposable, **players re-enter their info for each new game**.
  There is intentionally no account, no persistence of a person across games.
- **Host** is just a Player flagged as the game's creator (see T01); no special
  credentials beyond holding that game's host token.

**Forward-looking hook (future phase):** a later phase will introduce
**Google OAuth2** for structured identity. To avoid rework, the interim design keeps
identity behind a thin seam:

- A single **`PlayerIdentity` concept** (resolved from the join token today) is the
  only thing the rest of the system depends on — the game engine, scoring, and
  assignment never reference "how" a player authenticated.
- When OAuth2 arrives, the token-resolution layer is swapped/augmented to map a
  Google subject to a durable identity, without touching domain logic.
- No OAuth2 code, tables, or flows are built now; we only avoid designs that would
  make adding it later expensive (e.g. we don't hard-couple Player to a join token in
  the domain model — see T01).

## 4. Communication model

Two channels, both scoped per game:

1. **REST (request/response)** — T02. Used for commands and initial loads:
   create/start/join a game, upload a picture, submit a performance, confirm/
   volunteer, submit an evaluation or rating, post a comment or like, take a break,
   skip. Also used to fetch the current snapshot when a client (re)connects.

2. **WebSocket/STOMP (push)** — T03. Used for everything live: performer
   announcements, confirmation countdowns and timers, comment and like streams,
   score and ranking updates, reassignment (strikethrough) notifications, and
   performance state transitions.

**Snapshot + stream pattern:** on connect, a client fetches a **REST snapshot** of
the current game state, then **subscribes** to the game's topics for incremental
updates. This makes reconnection and late-join deterministic (details in T03).

**Role- and surface-aware delivery:** the backend controls what each subscriber
receives. For example, **judges do not receive comment events on their phones**,
while the TV and audience phones do (Business `06-comments-and-interaction.md`). The
authorization of subscriptions is a backend responsibility, not a frontend courtesy.

---

## 5. Game scoping

- Every domain object (Player, Performance, Rating, Evaluation, Comment, Like, queue
  entries) **belongs to exactly one Game**.
- All REST paths and WebSocket topics are **namespaced by game ID**, e.g.
  `/games/{gameId}/...` and topics like `/topic/games/{gameId}/ranking`.
- No cross-game reads or writes exist. This scoping is enforced at the API/subscription
  boundary (a token for game A cannot touch game B).

## 6. Configuration constants

Business rules reference tunable values. These are **externalized configuration**
(Spring `@ConfigurationProperties`), not hard-coded, so they can be adjusted without
code changes. Initial values:

| Constant | Meaning | Initial value | Source |
|----------|---------|---------------|--------|
| `MAX_DISPLAYED_PLAYER` | Max players shown per scoreboard page on the TV | **5** | Business 04 |
| `SCOREBOARD_PAGE_INTERVAL` | How long each scoreboard page is shown before cycling | ~**5 s** | Business 04 |
| `READY_CONFIRM_WINDOW` | Performer ready-confirmation window | **30 s** | Business 02/05 |
| `REPLACEMENT_OPEN_AFTER` | When volunteers may replace unconfirmed performers | **15 s** | Business 02/05 |
| `BREAK_DURATION` | "On a break" no-pick window | **15 min** | Business 05 |
| `PHYSICAL_CHALLENGE_JOB_INTERVAL` | Recurring check for injecting a challenge | ~**60 s** | Business 05 |
| `PHYSICAL_CHALLENGE_THRESHOLD` | Other-type performances since last challenge before injecting | **4** | Business 05 |
| `SUBJECTIVE_JUDGE_WEIGHT` / `SUBJECTIVE_AUDIENCE_WEIGHT` | Subjective score split | **0.85 / 0.15** | Business 04 |
| `LIKE_POINTS` | Points to a comment author per like | **1** (to confirm) | Business 04 |

Exact scoring point values (trivia correct/speed/penalty, etc.) are catalogued in
T05. **The complete, human-facing dial board** for every balance value — with meaning,
initial value, and the effect of changing it — is **T08 (Tuning & Game-Balance
Reference)**, the authoritative index for post-launch tuning. All values live under a
single `karaogui.*` config namespace.

## 7. Environments & deployment (high level)

- **Local development:** `docker-compose` runs PostgreSQL; the backend runs against
  it with Flyway applying migrations on startup; the two Angular apps run via the
  Angular dev server. Containerize the backend too for a one-command bring-up.
- **Configuration** via Spring profiles (`local`, etc.) and environment variables for
  DB connection and the config constants above.
- Detailed non-functional concerns (concurrency, observability, testing) are covered
  in T07.

## 9. Confirmed decisions

These were reviewed and locked in:

1. **Java 25 + Spring Boot 4.1** as the backend baseline.
2. **Gradle** as the build tool.
3. **STOMP-over-WebSocket** as the real-time transport (over SSE).
4. **Single Angular workspace with two apps** (`tv`, `phone`) sharing a contracts
   library.
5. **Both PostgreSQL and the backend are containerized** for a one-command local
   bring-up.
6. **Interim disposable per-game identity**; Google OAuth2 deferred to a later phase
   behind the identity seam described in §3.
